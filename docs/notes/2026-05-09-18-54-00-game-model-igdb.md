# Game model: IGDB API v4 + bundles + video linkage

Reference for modeling games (and game bundles like series / collections / genres) in pito and linking them to videos for analytics. Owned game library scope — not a full IGDB mirror.

Source: <https://api-docs.igdb.com/> (IGDB v4) and the canonical `Game` resource type at <https://github.com/DmitryScaletta/igdb-api-types>.

## API basics

- **Base URL**: `https://api.igdb.com/v4`
- **Auth**: Twitch OAuth 2.0 client-credentials. Two headers on every request: `Client-ID: <id>`, `Authorization: Bearer <token>`. Token from `POST https://id.twitch.tv/oauth2/token?client_id=...&client_secret=...&grant_type=client_credentials`. Lifetime ~60 days; cache and refresh.
- **Method**: `POST` everywhere. Query in the body as Apicalypse pseudo-SQL — every clause terminated by `;`.
- **Rate limit**: 4 req/s, max 8 in flight, 429 on overage.
- **Pagination**: `limit` (max 500 free, 3000 pro) + `offset`.
- **CORS**: blocked. Server-side only.
- **Sync model**: on-demand only (small library; no batch / cron sync needed). Each sync = read-modify-write the local game row, last-write-wins, manual edits get overwritten by re-sync.

## Apicalypse cheat-sheet

```
fields name, summary, cover.image_id, genres.name; 
where id = 119171; 
limit 1;
```

- Operators: `=`, `!=`, `>`, `<`, `>=`, `<=`, `~` (case-insensitive contains)
- Array operators: `[a,b]` (contains all), `(a,b)` (contains any), `{a,b}` (exact match)
- Logical: `&` (AND), `|` (OR)
- `fields *;` returns all top-level fields. `fields cover.*;` expands one ref. Multi-hop works: `involved_companies.company.name`.
- Search: `search "halo"; fields name; limit 10;` — relevance-sorted, no `where` allowed in same query.

### Time-to-beat detail

`game_time_to_beats` is a separate IGDB endpoint, one row per game keyed by `game_id`:

```
fields hastily, normally, completely; 
where game_id = 119171; 
limit 1;
```

Naming is unintuitive (`hastily` = main story, `normally` = main + extras, `completely` = 100%). Map them to local field names that make sense. Coverage is partial — store NULL when no row exists. Used as a rough length hint for scoring / future "what to cover next" decisions.

### Critic scores: posture

We surface critic scores as a **hint**, not as a primary signal. UI should display rating count alongside the score so a small sample is visible. No Metacritic-specific score is available via IGDB; `aggregated_rating` is the substitute.

### Steam / GOG / Epic

We store the external IDs IGDB gives us. We do **not** call Steam, GOG, or Epic APIs — explicit decision to keep the integration surface small. The IDs live on the model so a future feature can hydrate them without a migration.

To extract Steam ID from IGDB: query `external_games` with `where game = <igdb_id> & category = 1;`, read `.uid`. Same pattern for other stores; the `category` enum identifies the source.

## Things IGDB does NOT provide

Worth flagging so we don't go looking:

- **Metacritic-specific score**: Metacritic API is closed. `aggregated_rating` is the legal substitute — IGDB's own aggregation across critics, not labeled per-source.
- **Sales / revenue / units sold**: not in IGDB, not in any free API for the long tail. Field omitted from the model.
- **Per-platform pricing**: not in IGDB.
- **Steam reviews**: not in IGDB. (Available via Steam's `appreviews` endpoint if we ever add it — explicitly out of scope.)

## Local-only fields

These never come from IGDB. They're owned by the user and survive re-syncs.

| Field | Notes |
|---|---|
| `platform_owned` | Enum or FK to platform — which platform the user owns the game on. |
| `played_at` | Single date for v1. Can extend to a `played_sessions` table later if needed. |
| `notes` | Free-form. |
| `hours_of_footage` | Derived from joined videos (sum of durations). Cached column with manual override capability — `hours_of_footage_manual` nullable; if set, takes precedence over the derived sum. |
| `igdb_synced_at` | Bookkeeping. |

## Re-sync semantics

On user-triggered re-sync of a game:

1. Fetch the IGDB row by `igdb_id`.
2. Last-write-wins: every IGDB-sourced column is overwritten with the new value, even if locally edited.
3. Local-only fields (`platform_owned`, `played_at`, `notes`, `hours_of_footage_manual`) are untouched.
4. Update `igdb_synced_at`, `igdb_checksum`.

This is intentional and explicit per requirement: re-sync overrides local IGDB-field edits. If users want to preserve a manual edit on a synced field, they shouldn't re-sync until they want IGDB's version back.

## Bundles: series, collections, genres

A "bundle" is a named group of games used to link videos to a higher-level concept (e.g., "Zelda series", "soulslikes"). One model, multiple types.

```
bundle(
  id (local PK),
  bundle_type,            -- 'series' | 'collection' | 'genre' | 'custom'
  igdb_source_type,       -- nullable. 'franchise' | 'collection' | 'genre' | null for custom
  igdb_source_id,         -- nullable. The IGDB id of the source resource.
  name,
  composite_cover_path,   -- file path under composites/ (see below)
  composite_cover_checksum, -- detects when regen is needed
  created_at, updated_at
)

bundle_member(
  bundle_id,
  game_id,                -- local game.id
  position                -- ordering for display + composite layout
)
```

### IGDB sources for bundles

- **Series** = `franchises` endpoint. e.g., "The Legend of Zelda".
- **Collection** = `collections` endpoint. Smaller groupings, e.g., "Mario Kart".
- **Genre** = `genres` endpoint.

When a bundle is created from an IGDB source, `igdb_source_type` and `igdb_source_id` are populated and members are seeded from IGDB's listing. The user can then add or remove members — the bundle is not a strict mirror.

A `'custom'` bundle has no IGDB source — fully user-curated.

## Composite cover art

Bundles need a single image asset (ratatui can't composite at render time). Generated at sync time, regenerated on member changes.

- **Output**: 600×800 JPEG, 3:4 aspect ratio (matches IGDB cover ratio).
- **Storage**: `composites/{bundle_type}-{bundle_id}.jpg` under app data dir.
- **Source images**: each member's `cover_image_id` rendered at `t_cover_big` (227×320), downloaded once and cached.
- **Regen trigger**: `composite_cover_checksum` is the SHA-256 of the sorted list of member `cover_image_id` values plus the layout choice. If recomputed checksum differs from stored one, regenerate.
- **Layouts** by member count:
  - **2**: side by side, each 300×800
  - **3**: Netflix-style — one large left half (300×800), two stacked right (300×400 each)
  - **4**: 2×2 grid (300×400 each)
  - **5-9**: 3×3 grid, top-rated or most-recent first (200×267 each)
  - **10+**: same 3×3, truncated to 9 with a "+N" overlay on the bottom-right tile

Per-bundle override: `bundle.layout_override` nullable, lets us pick a different layout for a specific bundle later. Out of scope for v1 — default rule above.

Implementation: any standard image lib (Pillow / Rust `image` crate / imagemagick CLI). Synchronous from the user's POV is fine — small library, ~9 source images max, sub-second composite.

## Linking games & bundles to videos

Polymorphic link table — a video can be tagged with multiple games and/or bundles.

```
video_game_link(
  video_id,                        -- FK to local video
  link_type,                       -- 'game' | 'bundle'
  game_id,                         -- nullable, set when link_type='game'
  bundle_id,                       -- nullable, set when link_type='bundle'
  is_primary boolean,              -- for videos covering one main thing + side mentions
  created_at
)
```

Constraint: exactly one of `game_id` / `bundle_id` is set per row.

## Analytics integration: subscribers gained per video → game attribution

YouTube Analytics API v2 exposes `subscribersGained` and `subscribersLost` at video scope. From the analytics note:

```
ids=channel==MINE
filters=video==<video_id>
[no dimensions for lifetime, or dimensions=day for time-series]
metrics=subscribersGained,subscribersLost,views,estimatedMinutesWatched
```

This is already in `video_window_summary` and `video_daily` per the analytics note. No new YouTube query needed.

To translate to game attribution: SQL join `video_game_link` against `video_window_summary` (or `video_daily` for time-series), aggregate by `game_id` / `bundle_id`. Watch for double-counting when a video links to multiple games — use `is_primary` to weight or split:

- **Strict**: only count metrics for `is_primary=true` links. Lossy but unambiguous.
- **Even split**: divide each metric by the number of links on that video. Gives every linked game some credit.
- **Weighted**: requires manual weights per link. Out of scope.

Recommend even split as the default analytics view, with a toggle to switch to strict-primary.

This unlocks dashboards like:
- **Subs gained per game** (lifetime / 7d / 28d / 90d): which games drove channel growth.
- **Watch time per game**: which games kept viewers around.
- **Views per game per platform**: which platform's audience cares about which titles.
- **CTR per game** (using `videoThumbnailImpressionsClickRate`): which game thumbnails work.

These are all SQL queries against existing local tables — no new API calls.

## Future hooks (not implemented now)

- **Length-bucket scoring**: derive a length category from `ttb_main_seconds` (short / medium / long / epic) and use it in a "what to cover next" tool. Requires no model change beyond what's above.
- **Steam reviews**: Steam app ID is stored; a future feature can hydrate `steam_review_score` etc. without migration. Same for GOG / Epic.
- **MCP tools for game decisions**: e.g., "given this set of candidate games, recommend which to cover next based on past performance per genre / developer / length / rating." All inputs come from local tables once analytics are joined.
- **Per-bundle layout override**: add `layout_override` column to `bundle` if/when needed.

## Auth & rate-limit gotchas

1. Twitch credentials, not IGDB credentials. Sign up at `dev.twitch.tv`.
2. POST always, even for reads.
3. Apicalypse `;` terminator on every clause — easy to forget.
4. `category` and `status` fields on Game are deprecated; use `game_type` and `game_status`. Read both during transition, write logic against the new ones.
5. `first_release_date` is Unix **seconds** (not ms). Same for `created_at`, `updated_at`.
6. Search has no `where` clause — to search-and-filter (e.g. exclude editions), use the games endpoint with `where version_parent = null;` instead of the dedicated search endpoint, OR do two queries.
7. Cover URLs from `cover.url` default to `t_thumb` (90×90). Always build your own URL from `cover.image_id`.
