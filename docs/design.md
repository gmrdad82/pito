# pito — design system

The visual contract for pito's terminal aesthetic. Source of truth is the CSS:
`app/assets/tailwind/application.css` (tokens, component classes) and
`app/assets/tailwind/themes.css` (per-theme palettes, generated). This document
explains the rules those files encode — read it before any visual / component
work, and never paraphrase it into commit messages.

pito looks like a terminal: one monospace font, one size, square corners, no
hover theatrics, color and spacing (never size) for hierarchy.

## Typography

- **Font** — `--font-mono: "ui-monospace", monospace`. Everything is monospace.
  No proportional fonts anywhere.
- **One size** — `--text-base: 14px` (line-height `1.4`). `html` sets
  `font-size: var(--text-base)`; a base-layer rule `* { font-size: inherit }`
  propagates it to the whole tree. The `html` element selector outranks `*`, so
  it is the single source of truth for the base size.
- **No size creep** — never add a `font-size` declaration in CSS, and never use a
  Tailwind text-size utility (`text-xs` / `text-sm` / `text-lg` / `text-xl` / …).
  `text-base` is redundant — omit it. No `em` / `rem` / `px` font sizes, no
  sub-em shrinking for "secondary" text. Make secondary text **dim**
  (`text-fg-dim` / `text-fg-faded`), not small. Use weight, color, and spacing
  for hierarchy — never size.
- **Sole exception** — `.pito-start-screen__logo` (18px wordmark). It is the only
  place a `font-size` appears outside the base anchor.

> Note: an older "16px everywhere" phrasing predates the 14px base. The CSS
> (`--text-base: 14px`) is current and authoritative; the rule is "one size,"
> and that size is 14px.

## Theming

- **Token blocks** — `themes.css` is **generated by `rake pito:themes:export`**;
  do not hand-edit it. `:root` carries the default palette (tokyo-night); each
  named theme is a `[data-theme="<slug>"]` block declaring the same CSS custom
  property names with different hex values. ~19 themes ship, dark and light
  (ayu-dark/light/mirage, catppuccin-latte/mocha, dracula, github-dark/light,
  gruvbox-dark/light, nord, one-dark/light, solarized-dark/light, synthwave,
  tokyo-night, tomorrow, tomorrow-night).
- **`@theme` aliases** — `application.css` maps Tailwind color tokens onto the
  theme vars in one `@theme` block, e.g. `--color-root: var(--bg-root)`,
  `--color-fg-dim: var(--fg-dim)`, `--color-purple: var(--accent-purple)`,
  `--color-pito: var(--brand-pito)`. Tailwind v4 then derives the utilities
  (`bg-root`, `text-fg-dim`, `border-purple`, `text-pito`, …). Use these
  utilities, not raw hex.
- **Switching** — theme switching flips the `data-theme` attribute on `<html>`
  (persisted via `PATCH /settings/theme`). Because every block declares the same
  var names, the whole tree re-themes with no other change. The `/themes` chatbox
  command previews/applies via the `pito--theme-nav` sidebar.
- **Adding a theme** — add the palette to the export source and re-run
  `rake pito:themes:export`; never edit the generated `themes.css` by hand. A new
  block needs the full var set (same names) so every token resolves.

## Color palette

Every theme defines this exact set of CSS custom properties:

| Group       | Vars                                                          |
| ----------- | ------------------------------------------------------------ |
| Backgrounds | `--bg-root`, `--bg-surface`, `--bg-elevated`                 |
| Borders     | `--border-default`, `--border-faded`                         |
| Foreground  | `--fg-default`, `--fg-dim`, `--fg-faded`                     |
| Accents     | `--accent-purple`, `--accent-blue`, `--accent-cyan`, `--accent-green`, `--accent-yellow`, `--accent-orange`, `--accent-red` |
| Brand       | `--brand-pito` — **`#5170ff` ("pito blue"), constant across all themes** |

`--brand-pito` is the one fixed color: the same `#5170ff` in every block. It
drives the brand accent (the `.pito-shimmer` gradient sweeps
`--brand-pito → --accent-purple → --brand-pito`; the `data-accent="pito"`
segment bar). The 7 accents are the per-screen / per-message palette and are
theme-adaptive; heat-map components (`ScoreBar`, `TimeToBeat`) derive their
gradients from the active theme's accents via `color-mix(in oklch, …)` so they
read on all themes rather than being locked to literal hex.

### Message palette via `data-accent`

Components signal color through a **`data-accent` attribute** matched by static,
compiled CSS rules — never an inline `style=` and never a runtime-built Tailwind
arbitrary class. Examples in `application.css`:

- `.pito-segment__bar[data-accent="orange|red|yellow|purple|blue|surface|pito"]`
  — the 4px message segment bar color.
- `[data-accent="main|extras|completionist|footage"]` — TimeToBeat tick / legend
  colors.

Likewise the data-grid column count is driven by a `data-cols="N"` attribute with
precompiled `grid-template-columns` rules, not an arbitrary class.

### Copy-injected colors

Message copy can carry color (e.g. accent `text-*` classes resolved from
`Pito::Copy`). Because Tailwind v4's JIT only emits utilities it can see as
literal strings, any color class that originates in data/copy rather than in a
template **must be statically present in the source or safelisted** — otherwise
it is purged from the production build and the color silently disappears. Never
string-concat a Tailwind class from a variable.

## Component rules

- **Border-radius 0.** Square corners everywhere (scrollbar thumb, palettes,
  cards, snippets). No rounded UI.
- **No hover styling** for state/affordance. The one sanctioned hover is the
  `@handle` YouTube link underline (`.pito-channel-item__handle--link:hover`);
  do not add hover backgrounds, scales, or shadows elsewhere.
- **No inline `style=`.** Color and layout come from tokens, `data-accent` /
  `data-cols` attributes, and Tailwind utilities. The only inline `style` allowed
  is a genuinely per-record computed gradient (e.g. `TimeToBeat`'s per-game heat
  ramp from `#gradient_stops`) that cannot be precompiled.
- **Extract ViewComponents — no spaghetti.** All UI is a ViewComponent; never a
  raw `<button>` / `<div>` with inline classes in a view. When a utility cluster
  recurs 3+ times, extract a component (or `@apply` only if naming earns it).
  Reuse the shared components — `Pito::Table::KeyValueRowComponent`,
  `Pito::ScoreBarComponent`, `Pito::TimeToBeatComponent`,
  `Pito::Shell::ShimmerTextComponent`, `Pito::Event::MetaLineComponent`,
  `Pito::Event::SystemComponent` / `EnhancedComponent` — rather than reinventing
  them.
- **Logo font-size exception** — only `.pito-start-screen__logo` (18px) may set a
  font-size (see Typography).
- **Brand caps** — "pito" is lowercase in prose; "PITO" only appears in the logo
  art.
- **Interactivity is Stimulus** — controllers under
  `app/javascript/controllers/pito/`, one per file, loaded via importmap. No
  other JS framework, no inline `on*` handlers.
- **Server-pushed UI is Turbo Streams** — scrollback updates flow exclusively
  through `Pito::Stream::Broadcaster` on the `pito:conversation:<id>` stream
  (Turbo Frames for in-screen swaps; no polling). See `docs/architecture.md`.
