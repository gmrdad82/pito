## Concerns and suggestions

### Code review — non-blocking

1. **Filter param key inconsistency in inline comment (`_picker.html.erb:53`).**
   The new comment says
   `active filter chips (?starred=yes, ?connected=yes, etc.)`. The actual
   filter param is `star=yes` (no trailing "-red"). The spec at
   `spec/requests/channels_spec.rb:295` already calls this out via comment but
   uses the correct `star=yes` URL. Behavior is right; the inline ERB comment
   is stale. Trivial fix when next touched.
2. **`VideosController#sort_clause` duplicates the literal-string lookup
   instead of calling `sanitized_sort_key` / `sanitized_dir`.** Lines 152-156
   build `column = ALLOWED_SORTS[params[:sort]] || ALLOWED_SORTS[DEFAULT_SORT]`
   directly even though the controller exposes `sanitized_sort_key` for the
   same shape. Channels has the same micro-duplication. Safe (both branches
   read frozen allowlists), but the single-source-of-truth path is to call the
   `sanitized_*` helpers. Trivial future refactor.
3. **`docs/plans/beta/04-project-workspace/log.md` Wave 3 entry's hex values
   are out of date.** The log claims light tones `#f4f6f8 / #eef0f3 / #eef2f7`
   and dark `#2f3142 / #353748 / #2d3344`. The CSS actually ships
   `#f4f6f8 / #f0f2f5 / #f2f4f7` (light) and `#2f3142 / #34364a / #313346`
   (dark) per Wave 4. Doc-keeper update on next sweep. Reviewer is read-only
   on `docs/plans/`.
4. **`ChannelsController::ALLOWED_SORTS` exposes `"updated_at"` and
   `"channel_url"` even though no Wave-3.5 header links to them.** Same for
   `VideosController::ALLOWED_SORTS` — `created_at`, `updated_at`. Forward-
   looking allowlist members are fine and harmless, just noting that the
   `channel_url` and `updated_at` keys are not user-reachable from the index
   UI today.
5. **Notes pane still has the "syncing" lock branch above the bulk toolbar.**
   When `locked: true`, the pane renders the lock banner AND the bulk toolbar
   AND the table with always-on checkboxes. A user who ticks rows during a
   sync lock can still call `[delete N]`. Verify whether the architect wants
   the bulk surface gated behind `if !locked`. (Wave 3 brief asked for
   always-on but didn't mention lock interaction.) Safe today: the controller
   path enforces lock state when it processes the deletion. Worth a manual
   eyeball if the sync flow is exercisable in dev.
