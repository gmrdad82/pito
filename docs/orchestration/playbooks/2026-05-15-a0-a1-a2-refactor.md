## Manual test steps

Pre-flight (terminal). Run these before validating UI.

1. **Confirm credentials are in place.** Run
   `bin/rails credentials:edit --environment development`. The file must
   contain:

   ```yaml
   owner:
     username: gmrdad82
     password: <your-password>
   google_oauth:
     client_id: <youtube oauth client id>
     client_secret: <youtube oauth client secret>
     api_key: <youtube data api public key>
   voyage:
     development:
       api_key: <voyage key>
     test:
       api_key: <voyage key — optional>
     production:
       api_key: <voyage key — optional>
   ```

   The user mentioned adding a `cloudflare:` block and removing the legacy
   `owner.email` separately — confirm `owner.email` is gone and any new
   `cloudflare:` block is at the top level (sibling of `owner:` /
   `google_oauth:` / `voyage:`). The new `cloudflare:` block is informational
   for this validation; A0 / A1 / A2 don't read it.

   Repeat the relevant blocks for `--environment test`.

2. **Destructive reseed.** `bin/rails db:drop db:create db:migrate db:seed`.
   Confirm the seed output ends with "done!", lists the owner user
   (`user: gmrdad82 (id=...)`), seeds 6 platform rows, and does NOT print any
   "seeding project workspace sample…" / "seeding now playing collection…" /
   "channels…" / "videos…" lines.

3. **Restart the dev stack.** `bin/dev`. Wait for Puma + Sidekiq + Tailwind to
   come up. Confirm there is no boot-time crash from a missing credential
   (omniauth raises a loud error if `:google_oauth.client_id` / `:client_secret`
   are blank).

A0 — channel read-only mirror.

4. **Channel show is read-only.** Log in (you'll be bounced into TOTP setup —
   complete steps 13-15 first if it's the very first run after a reseed, then
   come back), then visit `/channels`. If the workspace is empty, connect a
   YouTube account via `[+ add Google account]` and let the OAuth pull mint a
   channel row. Open any channel via `/channels/:slug`. Confirm the heading-
   actions row shows `[ changes ]`, `[ sync ]`, `[ revoke ]`, `[ - ]` and does
   **NOT** show `[ e ]` / `[ edit ]`. View-source on the page, search for
   `channel_diff_banner` — absent.

5. **Removed routes 404.** With the channel slug from step 4, visit
   `/channels/<slug>/edit`, `/channels/<slug>/diff`, and
   `/channels/<slug>/preview` directly. All three return 404 / routing error
   pages (NOT an edit form, NOT a diff page, NOT a preview).

6. **Star toggle still works (HTML).** On `/channels`, select 2+ rows and open
   them with `[ open N ]` so panes render. In a pane, click `[ star ]` — the
   label flips to `[ unstar ]`, the page refreshes, the pane reflects the new
   state. Reload `/channels/<slug>` — the star state persisted. Click
   `[ unstar ]` to flip back.

7. **Star toggle still works (JSON, the CLI wire-format).** Pick a channel id N
   from `/channels`. In a terminal:

   ```
   curl -sS -X PATCH http://localhost:3027/channels/N/star.json \
     -H "Content-Type: application/json" \
     -d '{"channel":{"star":"yes"}}' | head
   ```

   Expect a 200 with a channel-detail JSON body where `"star"` reflects the
   toggle. Re-run with `{"channel":{"star":"no"}}` — also 200, toggled back.
   Re-run with `{"channel":{"star":"bad"}}` — expect 422 with an `errors` array.
   Re-run with `{"channel":{"star":"yes","title":"HACKED"}}` — expect 200;
   reload `/channels/<slug>` and confirm the title is **unchanged** (the removed
   field is silently ignored).

8. **Sync still pulls.** On `/channels/<slug>`, click `[ sync ]`. It should
   route to `/syncs/channel/:id` (no `?intent=diff_check` in the URL), render
   the confirmation screen, and on confirm enqueue a one-way `ChannelSync` job.
   Check `/sidekiq` (basic auth required) for the enqueued job, or refresh the
   channel show and confirm `last_synced_at` updated. No diff banner appears
   afterward.

9. **History still works.** Click `[ changes ]` — `/channels/<slug>/history`
   renders the read-only change-history table (or "no changes yet." empty
   state).

10. **Cron is gone.** In a terminal:
    `grep -E 'channel_diff_check|video_diff_check' config/sidekiq_cron.yml`.
    `video_diff_check_bulk` appears; `channel_diff_check` does not.


13. **OmniAuth still drives `[ + add Google account ]`.** On `/channels`, click
    the `[ + add Google account ]` button. The flow should hand off to Google's
    OAuth consent screen (or, in dev, to your mock if you have one). The

A2 — User auth refactor (start from a fresh seed if you can — step 2 above
already did this).

14. **Login form uses `username`.** On a logged-out browser, visit `/login`. The
    first form input is a `username` text field (not an `email` field). The lead
    paragraph reads "sign in with your pito account. forgot your password?
    [reset password]" and the `[ reset password ]` link points at
    `/password/reset`.

15. **First login forces TOTP setup.** Submit `gmrdad82` + the credentials
    password from step 1. You are redirected to `/settings/security/totp` (the
    TOTP enrollment landing page). Attempt to visit `/channels`, `/videos`,
    `/settings`, `/projects` — every one of them bounces back to
    `/settings/security/totp` with the alert "set up two-factor authentication
    to continue.".

16. **Complete TOTP enrollment.** On the enrollment page, click through
    `[ generate code ]` to mint the seed. Scan the QR (or paste the base32 seed)
    into your authenticator app. **Copy the 10 backup codes to a safe place —
    you'll need one in step 19.** Confirm a 6-digit code from your
    authenticator. After confirmation, the gate releases: visit `/channels` — it
    loads.

17. **Log out and back in.** Click `[ log out ]`. Log in again with `gmrdad82` +
    password. This time you are routed to `/login/totp` and must enter a current
    6-digit code. Enter one — you reach the app.

18. **Reset password via 2FA (live TOTP code).** Log out. On `/login`, click
    `[ reset password ]`. On `/password/reset`, enter `gmrdad82` + a current
    6-digit code from your authenticator. You land on `/password/reset/edit`.
    Enter a new password twice and submit. You are redirected to `/login` with
    the flash "password reset. log in with your new password." — and you are NOT
    auto-logged-in. Log in with `gmrdad82` + the NEW password, pass the TOTP
    challenge, reach the app. Confirm the OLD password no longer works (generic
    `login failed.`).

19. **Reset password via 2FA (backup code, single-use).** Log out. On `/login`,
    click `[ reset password ]`. On `/password/reset`, enter `gmrdad82` + **one
    of the backup codes** you saved in step 16. The flow succeeds and you land
    on the set-password page; complete the reset. Log in with the new password.

    Then attempt the reset flow AGAIN with the SAME backup code you just used —
    expect a generic `reset failed.` (the backup code was consumed by the
    previous use).

20. **No account-existence oracle.** On `/login`, try a nonexistent username
    (`doesnotexist`) + any password — the failure response should be
    byte-identical (status, body, response time within tolerance) to a
    wrong-password attempt for `gmrdad82`. On `/password/reset`, try
    `doesnotexist` + any code — the generic `reset failed.` is identical to
    submitting `gmrdad82` + a wrong code.

21. **Throttle the reset endpoint.** From a single client, POST to
    `/password/reset` 6 times within a minute:

    ```
    for i in 1 2 3 4 5 6; do
      curl -sSo /dev/null -w "%{http_code}\n" -X POST \
        http://localhost:3027/password/reset \
        -d "username=gmrdad82&code=000000"
    done
    ```

    Expect five 200/422 responses and the 6th to return a 429 with the generic
    `reset failed.` body.

22. **Operator TOTP-reset rake task — happy path.** In a separate terminal:
    `bin/rails pito:user:reset_totp[gmrdad82]`. Confirm the printed line: "TOTP
    reset for gmrdad82 — sessions revoked, backup codes cleared. They will be
    forced through TOTP setup on next login." Then, in the browser, refresh any
    tab — your session is dead, you are bounced to `/login`. Log in with
    `gmrdad82` + password — the mandatory-2FA gate forces you through fresh
    enrollment (re-do step 16).

23. **Operator TOTP-reset rake task — sad path.** Run
    `bin/rails pito:user:reset_totp[nosuchuser]`. The task prints
    `user not found: nosuchuser` to stderr and exits non-zero (`echo $?` after
    the task should print `1`).
