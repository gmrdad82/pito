# Phase 20 — Friendly URLs (App-Wide)

> Drop integer IDs from the address bar. Every user-facing resource URL becomes
> a slug-shaped path. Mechanism is the `friendly_id` gem, applied uniformly
> across web, MCP, and the `pito` CLI. Date: 2026-05-10. User directive on the
> same date: "Address app wide implementation of nicer urls. I don't want to see
> ids in the addressbar anymore."

---

## Goal

Replace integer-ID URLs (`/projects/42`, `/games/97`, `/channels/3`) with
slug-shaped URLs (`/projects/my-summer-game-essay`, `/games/celeste`,
`/channels/UCxyz123abc`) across every user-facing surface that exposes a
resource — web routing, MCP tool inputs, and the `pito` CLI.

Why it matters for the project: integer IDs leak ordering and volume, read worse
in URLs and bookmarks, and break the project's "feels like a hand-built tool"
aesthetic. They also make sharing URLs (with the user, with oneself across
devices, in conversation logs) needlessly opaque. A slug-based URL is
self-describing: `/projects/celeste-retrospective` says what it points at;
`/projects/42` does not.

Who uses it: every actor that constructs or follows a URL — the user clicking
through the web app, Claude Mobile / Desktop hitting MCP tools, the `pito` CLI
surfaces that link to web pages, and any future automation that reads URLs from
the database.

The mechanism is shared (the `friendly_id` gem, configured once) so each new
resource that gets sluggable in future phases inherits the pattern with two or
three lines of code rather than reinventing slug generation, conflict suffixing,
history, transliteration, length capping, and edge handling.

---

## Decisions locked by the master agent (do not relitigate)

1. **Use the `friendly_id` gem.** Battle-tested, integrates with Rails routing,
   supports history (renamed-slug redirect), supports slug-conflict suffixing
   out of the box. A hand-rolled `Sluggable` concern duplicates work the gem
   already does.
2. **Slug column only when there is no existing natural identifier.** Channel,
   Video, Game, Footage already have URL-safe unique columns and reuse them via
   `friendly_id` configured to `:column_name`. Project, Bundle, Collection,
   MilestoneRule get a new `slug` column with a unique index, populated from
   `name` via `friendly_id :name, use: :slugged`.
3. **History module enabled on user-renameable resources** (Project, Bundle,
   Collection, MilestoneRule) so old slugs 301 to current after a rename.
   Disabled on identifier-style resources (Channel, Video, Game, Footage) —
   those slugs are externally owned and won't change.
4. **Backwards compat: `Model.friendly.find(param)` accepts both slug and
   integer ID.** Existing bookmarks, log lines, MCP-saved URLs, and CLI args
   containing integer IDs keep resolving. The integer-ID path is permitted but
   `to_param` always returns the slug, so any redirect or re-render emits the
   slug form.
5. **MCP and CLI accept both slug and integer ID at the boundary.** Tests cover
   both inputs per tool / per CLI subcommand.
6. **CalendarEntry: skipped for now.** Calendar surfaces don't expose entry IDs
   in the address bar in a way that benefits from slugging
   (`/deletions/calendar_entry/:id` is a confirmation pipeline, not a
   bookmarkable resource page). Revisit when Video Workflow Features (Phase 10)
   ships its own dedicated calendar UI.
7. **Doorkeeper applications keep integer IDs.** Application IDs in URLs are
   sensitive surfaces (paired with secrets, tokens, revoke flows); exposing
   slugs would invite guessing-by-name. Out of scope.
8. **No per-User slugs.** No public profile pages exist (single-install
   multi-user); there is nothing to URL-address.
9. **Settings sub-pages already singular.** `/settings/general`,
   `/settings/tokens`, etc. are not per-instance resources and need no change.

### New `slug` column migrations

One migration per resource that needs a new slug column. Each migration:

1. Adds `slug :string` (nullable initially).
2. Backfills existing rows in the same migration via `set_friendly_id` on each
   record (use `find_each` + `save!(validate: false)` to avoid re-running other
   validations).
3. Adds a unique index on `slug`.
4. Sets `slug NOT NULL` after backfill.

Files:

### Controller changes

Every controller that currently calls `Model.find(params[:id])` (or
`params[:foo_id]`) for a slugged resource switches to
`Model.friendly.find(...)`. The `:finders` friendly_id module makes both slug
and integer-ID lookups work via the same call.

Controllers to audit and update (resource → file):

- Channel — `app/controllers/channels_controller.rb` and any nested controllers
  that scope by `channel_id` (e.g., videos under a channel, the `channels`
  workspace pane resolver).
- Video — `app/controllers/videos_controller.rb` and any nested.
- Game — `app/controllers/games_controller.rb`.
- Project — `app/controllers/projects_controller.rb` plus any nested (project
  notes, project footage, project bundles, project collections).
- Footage — `app/controllers/footage_controller.rb` (or wherever footage show /
  edit lives).
- Bundle — `app/controllers/bundles_controller.rb`.
- Collection — `app/controllers/collections_controller.rb`.
- MilestoneRule — `app/controllers/milestone_rules_controller.rb` (or its
  current location under settings).
- Note — `app/controllers/notes_controller.rb` (use
  `Note.find_by!(path: params[:id])`).

The controllers must also redirect integer-ID hits to the slug URL when the
request is a `GET` rendering the resource. Pattern:

```ruby
def show
  @project = Project.friendly.find(params[:id])
  if request.path != project_path(@project)
    redirect_to project_path(@project), status: :moved_permanently
    return
  end
  # ...
end
```

A shared `before_action` (e.g., a thin `FriendlyRedirect` concern) is acceptable
but not mandatory; the spec leaves implementation to the rails-impl agent.
Either pattern is fine as long as integer-ID GETs redirect to the canonical slug
URL.

### Routes

`config/routes.rb` does **not** change in declaration shape. `resources :foo`

generated URL becomes a slug URL automatically.

Verify (no change required, just a note for the implementer): no route
references rely on numeric `:id` constraints. If any do (e.g.,
`constraints: { id: /\d+/ }`), they must be removed for slugged resources.

## Test sweep (per the user's "fully specced" directive — exhaustive)

The pyramid (`docs/agents/architect.md` §D) applies. Concretely:

### Shared / cross-cutting

- A spec under `spec/lib/friendly_id_setup_spec.rb` (or an equivalent suite)
  verifies the `friendly_id_slugs` table exists and that the `:history` module
  is enabled by checking that `Project.friendly_id_config.uses?(:history)`
  returns true, and that Channel / Video / Game / Footage do NOT include
  `:history`.

### Per slugged resource — Project / Bundle / Collection / MilestoneRule (slug from `name`)

For each of the four `:slugged` + `:history` resources,
`spec/models/<model>_spec.rb` adds:

- Slug is generated on `create` from `name`.
- Two records with the same `name` get distinct slugs (`-2`, `-3` suffix
  resolution).
- `to_param` returns the slug, not the integer ID.
- `Model.friendly.find(slug)` returns the record.
- `Model.friendly.find(id.to_s)` returns the record (backwards-compat).
- `Model.friendly.find(id)` (integer) returns the record (backwards-compat).
- Renaming the record via `update!(name: "New name")` regenerates the slug.
- After rename, `Model.friendly.find(<old slug>)` still returns the record
  (history module).
- `name` set to an empty string falls back to a non-empty default (e.g., the
  integer ID, or a typed prefix like `project-<id>`).
- Unicode in `name` is transliterated (e.g., `"Café"` → `"cafe"`).
- Very long `name` (>255 chars) is truncated to a slug under the column limit.
- Edge characters in `name` (slashes, leading/trailing whitespace, multiple
  separators) are stripped.

### Per slugged resource — Channel / Video / Game / Footage (slug from natural column)

For each of the four identifier-style resources, `spec/models/<model>_spec.rb`
adds:

- `to_param` returns the natural identifier column value (or the derived
  `url_slug` method's value), not the integer ID.
- `Model.friendly.find(slug)` returns the record.
- `Model.friendly.find(id.to_s)` returns the record (backwards-compat).
- `Model.friendly.find(id)` (integer) returns the record (backwards-compat).
- Two records with the same natural identifier raise a uniqueness error (the
  underlying column already has a unique index — verify it's enforced).
- For Channel: `url_slug` extracts the UC-ID portion correctly from a variety of
  `channel_url` shapes (e.g., `https://youtube.com/@Handle`,
  `https://youtube.com/channel/UCxyz`, with and without trailing slash).
- For Footage: `url_slug` strips the path / extension cleanly and parameterizes
  safely (no slashes, no spaces).

### Request specs — every controller for a slugged resource

For each controller (`spec/requests/<resource>_spec.rb`):

- `GET /foos/<slug>` returns 200 and renders the resource.
- `GET /foos/<integer-id>` returns 301 redirecting to `/foos/<slug>`.
- `GET /foos/<old slug after rename>` returns 301 redirecting to
  `/foos/<new slug>` (only for resources with `:history`).
- `GET /foos/does-not-exist` returns 404.
- Nested routes (e.g., `/projects/<slug>/footage/<footage-slug>`) resolve both
  segments correctly.
- `PATCH /foos/<slug>` (or POST) accepts the slug as the route param and updates
  the record.
- `DELETE /deletions/<type>/<slug>` (per the bulk-as-foundation framework)
  accepts the slug; the existing comma-separated `:ids` segment becomes a
  comma-separated `:keys` segment that accepts slug or integer ID per element.
  Verify single-record and multi-record deletions still resolve.
- `POST /syncs/<type>/<slug>` (Channel sync framework) accepts the slug. Same
  check.

### MCP tool specs

For every MCP tool that accepts a slugged resource identifier
(`spec/mcp/tools/<tool>_spec.rb`):

- Tool resolves the record when `id` argument is a slug.
- Tool resolves the record when `id` argument is the integer ID as a string.
- Tool returns the documented "not found" error envelope when the identifier
  matches nothing.
- For bulk tools (`delete_records`, `sync_records`), the comma-separated
  identifier list accepts a mix of slugs and integer IDs.

### CLI specs

For every `pito` subcommand argument that previously took an integer ID, the
Rust crate's test suite (under `extras/cli/`) covers:

- The new `--<resource>` flag accepts a slug.
- The new `--<resource>` flag accepts an integer-ID string.
- The legacy `--<resource>-id` alias still works (prints a deprecation notice on
  stderr; resolves identically).

### System specs (selective, per pyramid §10)

One thin system spec covers the canonical journey:

- Visit `/projects/<integer id>` → automatically redirected to
  `/projects/<slug>`.
- Edit the project name, save → redirected to the new slug URL.
- Manually visit the OLD slug URL → 301 redirected to the new slug URL (history
  module).
- Manually visit a never-existed slug URL → 404.

That single spec exercises the entire user-visible contract. No further system
specs needed.

### Routing specs

Routing logic is unchanged (`resources :foo` still applies). One thin routing
spec per slugged resource is acceptable but not mandatory; the implementation
agent decides based on whether any custom route constraints were touched.

## Open questions

1. **Channel `url_slug` extraction logic.** Channel URLs come in three shapes
   (`/@Handle`, `/channel/UCxyz`, `/c/CustomName`). The spec assumes UC-ID
   extraction is the right slug. If a channel was created with a `/@Handle` URL
   and we never fetched the UC ID (Phase 7 / 8 work hasn't landed), the channel
   has no UC ID and falls back to — what? The handle? An auto-generated
   `channel-<id>` slug? Master agent should confirm before rails-impl picks one.
2. **Footage `url_slug` collisions.** Two footage files in different
   sub-directories can share a basename (`/a/clip.mp4` and `/b/clip.mp4`).
   Reusing `local_path` basename as the slug guarantees collisions. Options:
   include parent directory in the slug, append a hash suffix, or fall back to a
   `:slugged` column. Master agent should confirm before rails-impl picks one.
3. **Note URL shape.** `Note#path` may contain slashes
   (`projects/foo/notes/bar.md`). A slash-bearing slug doesn't fit `/notes/:id`
   cleanly. Options: route as a glob (`/notes/*path`), strip slashes for a flat
   slug, or keep notes accessed only nested under their owning project. Master
   agent should confirm before rails-impl picks one.
4. **Bulk URL segment naming.** The bulk-as-foundation framework uses
   `/<action>s/:type/:ids`. Once IDs become slugs, `:ids` is a misnomer.
   Renaming to `:keys` is correct but breaks any docs / saved URLs that
   reference `:ids`. Confirm whether the rename is in scope this phase or
   deferred to a docs-only follow-up.
5. **Friendly_id version pin.** The spec assumes `~> 5.5`. Confirm the actual
   current major; pin precisely once the implementation agent resolves the gem
   index.
6. **Slug column max length.** `friendly_id` defaults to 240 chars. Some
   resources may want a tighter cap (e.g., Project: 80) for clean URLs. Defer to
   rails-impl unless master agent wants a cap stated up front.

## Master agent decisions (2026-05-10)

Master agent has resolved every open question per the autonomy rule.
Implementation agent treats these as the contract.

1. **Channel `url_slug` fallback when UC ID isn't yet fetched.** Fall back to
   the integer ID with a prefix: `channel-<id>`. Once the channel sync populates
   the UC ID (post-OAuth, Phase 7 / 8 work), the canonical slug becomes the
   UC-id. The history module redirects the old `channel-<id>` URL to the new
   UC-id URL. Add a model-level method `Channel#computed_slug` that returns the
   UC-id if present, else `"channel-#{id}"`.
2. **Footage basename collision policy.** Append `-2`, `-3`, etc. as a numeric
   suffix on collision. Mirror the `friendly_id` default candidate-resolution
   shape (basename, basename-2, basename-3, …).
3. **Note URL shape with slash-bearing paths.** Use `path` directly with URL
   encoding. Routes use a `/notes/*path` glob constraint so the path segment can
   contain slashes. `Note#to_param` returns `path` verbatim; the router and
   controllers handle decoding.
4. **Bulk URL `:ids` vs `:keys` rename scope.** Drop the rename. Keep the `:ids`
   parameter name; it accepts both integer IDs and slugs interchangeably.
   Backwards compatibility with existing docs and saved URLs is preserved. No
   router or controller signature changes for the bulk-as-foundation framework.
5. **friendly_id version pin.** `~> 5.5` (latest 5.x). Implementation agent pins
   this exact constraint in the `Gemfile`.
6. **Per-resource slug max length.** 80 chars by default. Configurable via a
   `slug_limit` declaration on the model when a specific resource needs more.
   Truncation strips trailing word boundaries cleanly (cut at the last
   whitespace / hyphen before the limit, never mid-word).
